In Part 3, we created a WebLogic domain with 9 servers across two hosts. The domain works perfectly‚Äîuntil someone reboots a server. In this part, we'll configure systemd services for automatic startup, ensuring your P6 EPPM cluster survives server reboots without manual intervention.

The challenge with multi-host WebLogic clusters is startup order: Node Manager must start before Admin Server, Admin Server before Managed Servers, and the secondary host must wait for the primary host's Admin Server. We'll solve this with systemd dependencies and a clever wait loop.

[et_pb_image src="https://integrationfaces.com/wp-content/uploads/2025/12/p6-eppm-25.12-architecture-diagram.png" alt="P6 EPPM Architecture - Service Management" title_text="P6 EPPM 25.12 Architecture" align="center" _builder_version="4.24.0" _module_preset="default" max_width="900px" module_alignment="center"][/et_pb_image]

What We're Building

By the end of this part, your WebLogic cluster will:

  • Start automatically after any server reboot
  • Maintain proper sequence: Node Manager ‚Üí Admin Server ‚Üí Managed Servers
  • Handle cross-host dependencies: prmapp02 waits for prmapp01's Admin Server
  • Provide clean service management: start, stop, status via systemctl
  • Log to journald for easy troubleshooting

Startup Architecture

Host Service Depends On Starts
prmapp01 Node Manager network.target Port 5556
Admin Server Node Manager Port 7001
Managed Servers Admin Server Ports 7010-7040
prmapp02 Node Manager network.target Port 5556
Managed Servers Node Manager + Admin Server (remote) Ports 7010-7040

Prerequisites

  • Part 3 completed: WebLogic domain created with all 9 servers
  • Domain tested: All servers start manually via WebLogic Console
  • SSH access to both prmapp01 and prmapp02
  • Root access for systemd configuration
  • WebLogic admin password ready

Important: Before proceeding, verify your domain is working by starting all servers manually through the WebLogic Console. This ensures any issues you encounter are related to auto-start configuration, not the underlying domain.

Step 1: Download the Scripts

We'll use the scripts from the GitHub repository. These have been tested in production environments and adapted for our blog series lab configuration.

ls -la

You should see service files, WLST scripts, and utility scripts for both hosts.

Step 2: Create boot.properties Files (prmapp01)

WebLogic servers need credentials to start unattended. The boot.properties file stores username and password, which WebLogic encrypts on first startup.

On prmapp01 as oracle user:

[et_pb_code _builder_version="4.24.0" _module_preset="default"] 
# Make script executable
chmod +x create-boot-properties.sh

# Run as oracle user (use single quotes for special characters in password)
./create-boot-properties.sh weblogic 'YourSecurePassword'
[/et_pb_code]

The script auto-detects prmapp01 and creates boot.properties for:

  • AdminServer
  • p6web_ms1
  • p6ws_ms1
  • p6tm_ms1
  • p6cc_ms1

Verify the files were created:

[et_pb_code _builder_version="4.24.0" _module_preset="default"] 
ls -la /u01/app/weblogic/user_projects/domains/eppm_domain/servers/*/security/boot.properties
[/et_pb_code]

You should see 5 files, each with permissions 600 (readable only by owner).

Step 3: Install Node Manager Service (prmapp01)

The Node Manager service is the foundation‚Äîall other services depend on it. Let's examine the service file before installing:

[et_pb_code _builder_version="4.24.0" _module_preset="default"] 
# View the service file
cat prmapp01/weblogic-nodemanager.service
[/et_pb_code]

Key configuration points:

  • Type=simple: Node Manager runs in foreground, systemd tracks the process
  • After=network.target: Waits for network to be available
  • Restart=on-failure: Automatically restarts if it crashes
  • TimeoutStartSec=300: Allows up to 5 minutes to start

Install the service:

[et_pb_code _builder_version="4.24.0" _module_preset="default"] 
# Copy service file (as root)
sudo cp prmapp01/weblogic-nodemanager.service /etc/systemd/system/
sudo chmod 644 /etc/systemd/system/weblogic-nodemanager.service

# Reload systemd to recognize the new service
sudo systemctl daemon-reload

# Enable auto-start on boot
sudo systemctl enable weblogic-nodemanager

# Verify it's enabled
systemctl is-enabled weblogic-nodemanager
[/et_pb_code]

Step 4: Install Admin Server Service (prmapp01)

The Admin Server service has dependencies on Node Manager:

[et_pb_code _builder_version="4.24.0" _module_preset="default"] 
# View the service file
cat prmapp01/weblogic-adminserver.service
[/et_pb_code]

Key dependency directives:

  • After=weblogic-nodemanager.service: Starts after Node Manager
  • Requires=weblogic-nodemanager.service: Won't start if Node Manager isn't running
  • BindsTo=weblogic-nodemanager.service: Stops if Node Manager stops

Install the service:

[et_pb_code _builder_version="4.24.0" _module_preset="default"] 
# Copy and enable
sudo cp prmapp01/weblogic-adminserver.service /etc/systemd/system/
sudo chmod 644 /etc/systemd/system/weblogic-adminserver.service
sudo systemctl daemon-reload
sudo systemctl enable weblogic-adminserver
[/et_pb_code]

Step 5: Install Managed Servers Service (prmapp01)

The Managed Servers service is different‚Äîit uses Type=oneshot because the WLST script completes after starting the servers:

[et_pb_code _builder_version="4.24.0" _module_preset="default"] 
# View the service file
cat prmapp01/weblogic-managedservers.service
[/et_pb_code]

Key features:

  • Type=oneshot: Runs WLST script once, then exits
  • RemainAfterExit=yes: Keeps service "active" after script completes
  • ExecStartPre: Wait loop checks Admin Server port before starting

The wait loop is critical‚Äîit polls http://prmapp01:7001 every 5 seconds for up to 5 minutes, ensuring Admin Server is ready before attempting to start managed servers.

Install WLST scripts and service:

[et_pb_code _builder_version="4.24.0" _module_preset="default"] 
# Copy WLST scripts to oracle home directory
cp prmapp01/start-managed-servers.py /home/oracle/
cp prmapp01/stop-managed-servers.py /home/oracle/
chmod 750 /home/oracle/*.py

# IMPORTANT: Update password in WLST scripts
sed -i "s/CHANGE_ME/YourSecurePassword/" /home/oracle/start-managed-servers.py
sed -i "s/CHANGE_ME/YourSecurePassword/" /home/oracle/stop-managed-servers.py

# Verify password was updated
grep admin_password /home/oracle/start-managed-servers.py

# Copy and enable service
sudo cp prmapp01/weblogic-managedservers.service /etc/systemd/system/
sudo chmod 644 /etc/systemd/system/weblogic-managedservers.service
sudo systemctl daemon-reload
sudo systemctl enable weblogic-managedservers
[/et_pb_code]

Step 6: Configure prmapp02 (Secondary Host)

Now repeat the process on prmapp02, but note these differences:

  • No Admin Server service‚ÄîAdmin Server only runs on prmapp01
  • Managed Servers wait for remote Admin Server‚Äîthe wait loop checks prmapp01:7001
  • Longer timeout‚Äî900 seconds instead of 600 to allow for prmapp01 boot time

On prmapp02:

[et_pb_code _builder_version="4.24.0" _module_preset="default"] 
# Clone repository (or copy files from prmapp01)
cd /tmp
git clone https://github.com/integration-faces/zero-to-enterprise-p6eppm.git
cd zero-to-enterprise-p6eppm/part4-service-management

# Create boot.properties (as oracle user)
chmod +x create-boot-properties.sh
./create-boot-properties.sh weblogic 'YourSecurePassword'

# Copy WLST scripts
cp prmapp02/start-managed-servers.py /home/oracle/
cp prmapp02/stop-managed-servers.py /home/oracle/
chmod 750 /home/oracle/*.py
sed -i "s/CHANGE_ME/YourSecurePassword/" /home/oracle/*.py

# Install service files (as root) - NOTE: Only 2 services!
sudo cp prmapp02/weblogic-nodemanager.service /etc/systemd/system/
sudo cp prmapp02/weblogic-managedservers.service /etc/systemd/system/
sudo chmod 644 /etc/systemd/system/weblogic-*.service
sudo systemctl daemon-reload

# Enable services
sudo systemctl enable weblogic-nodemanager
sudo systemctl enable weblogic-managedservers
[/et_pb_code]

Step 7: Test Manual Service Startup

Before testing reboot persistence, let's verify the services work manually. Start with prmapp01:

[et_pb_code _builder_version="4.24.0" _module_preset="default"] 
# Start Node Manager
sudo systemctl start weblogic-nodemanager
systemctl status weblogic-nodemanager

# Wait a few seconds, then start Admin Server
sleep 10
sudo systemctl start weblogic-adminserver

# Watch Admin Server startup (Ctrl+C when ready)
journalctl -u weblogic-adminserver -f

# Verify Admin Server is responding
curl -s http://prmapp01:7001 | head -5

# Start Managed Servers
sudo systemctl start weblogic-managedservers

# Watch managed servers startup
journalctl -u weblogic-managedservers -f
[/et_pb_code]

Then on prmapp02 (Admin Server on prmapp01 must be running first):

[et_pb_code _builder_version="4.24.0" _module_preset="default"] 
# Test connectivity to prmapp01 Admin Server
curl -s http://prmapp01:7001 | head -5

# Start services
sudo systemctl start weblogic-nodemanager
sudo systemctl start weblogic-managedservers

# Watch startup
journalctl -u weblogic-managedservers -f
[/et_pb_code]

Step 9: Test Reboot Persistence

The ultimate test‚Äîreboot both servers and verify everything comes back automatically.

Graceful shutdown first:

[et_pb_code _builder_version="4.24.0" _module_preset="default"] 
# Stop prmapp02 first
ssh prmapp02 'sudo systemctl stop weblogic-managedservers'
ssh prmapp02 'sudo systemctl stop weblogic-nodemanager'

# Then prmapp01
sudo systemctl stop weblogic-managedservers
sudo systemctl stop weblogic-adminserver
sudo systemctl stop weblogic-nodemanager
[/et_pb_code]

Reboot prmapp01:

[et_pb_code _builder_version="4.24.0" _module_preset="default"] 
sudo reboot
[/et_pb_code]

Wait 5-10 minutes for prmapp01 to fully boot and start all services. Monitor from another machine:

[et_pb_code _builder_version="4.24.0" _module_preset="default"] 
# From your workstation or prminfra01, poll until Admin Server responds
while ! curl -s http://prmapp01:7001 >/dev/null; do
  echo "Waiting for prmapp01 Admin Server..."
  sleep 10
done
echo "prmapp01 Admin Server is ready!"
[/et_pb_code]

Reboot prmapp02:

[et_pb_code _builder_version="4.24.0" _module_preset="default"] 
ssh prmapp02 'sudo reboot'
[/et_pb_code]

Wait 5 minutes, then verify all servers are running via WLST or the WebLogic Console.

Check startup timestamps to confirm auto-start worked:

[et_pb_code _builder_version="4.24.0" _module_preset="default"] 
# On prmapp01
systemctl show weblogic-nodemanager -p ActiveEnterTimestamp
systemctl show weblogic-adminserver -p ActiveEnterTimestamp
systemctl show weblogic-managedservers -p ActiveEnterTimestamp
[/et_pb_code]

The timestamps should show proper sequence: Node Manager started first, then Admin Server, then Managed Servers.

Quick Reference: Service Commands

Action Command
Check status systemctl status weblogic-*
Start all (prmapp01) sudo systemctl start weblogic-nodemanager weblogic-adminserver weblogic-managedservers
Stop all (prmapp01) sudo systemctl stop weblogic-managedservers weblogic-adminserver weblogic-nodemanager

Troubleshooting

Node Manager won't start:

  • Check port 5556 isn't in use: ss -tlnp | grep 5556
  • Verify script path: ls -l /u01/app/weblogic/user_projects/domains/eppm_domain/bin/startNodeManager.sh
  • Check logs: journalctl -u weblogic-nodemanager -n 50

Admin Server won't start:

  • Verify Node Manager is running: systemctl status weblogic-nodemanager
  • Check boot.properties exists: ls -l /u01/app/weblogic/user_projects/domains/eppm_domain/servers/AdminServer/security/boot.properties
  • Check logs: journalctl -u weblogic-adminserver -n 50

Managed Servers won't start:

  • Test Admin Server connectivity: curl http://prmapp01:7001
  • Verify password in WLST scripts: grep admin_password /home/oracle/start-managed-servers.py
  • Check boot.properties for each managed server
  • Check logs: journalctl -u weblogic-managedservers -n 100

prmapp02 times out waiting for Admin Server:

  • Ensure prmapp01 boots and starts Admin Server first
  • Check firewall: sudo firewall-cmd --list-ports | grep 7001
  • Add firewall rule if needed: sudo firewall-cmd --permanent --add-port=7001/tcp && sudo firewall-cmd --reload
  • Increase timeout in service file (change {1..60} to {1..120} for 10 minutes)
